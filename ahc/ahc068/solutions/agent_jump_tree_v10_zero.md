# jump_tree_v10_zero

`agent_jump_tree_v9.rs`を基準に、壁が0本の入力だけを固定作業量の
C-R-C（列→行→列）ルータへ分岐する候補。

## 分岐

- `wall_count == 0`: 二部多重グラフの辺彩色で各カードの中継行を決め、
  60本のラインを等長隣接ブロック交換のビームで整列する。
- `wall_count > 0`: V9の局所探索、チェックポイント、長距離ジャンプ木補完を
  変更せず実行する。
- ゼロ壁分岐ではV9候補を並行生成しない。したがって両エンジンの実行時間は
  加算されない。

## 機械非依存の作業量

ゼロ壁ビームは`Instant`を参照しない。以下の固定上限だけで停止する。

- beam width: 200
- depth: 14
- 1ラインの展開試行: 240,000以下
- 1入力全体の展開試行: 14,400,000以下

評価値は`100 * breakpoints + displacement`、状態キーは20要素を5 bitずつ
格納する`u128`。同点時は状態キーで順序を確定するため、ハッシュの乱数シードや
マシン速度で選択結果が変化しない。

固定上限でビームがラインを完成できなかった場合は、そのラインだけ隣接交換で
厳密に完成させる。このため3相の不変条件は維持され、最終盤面は必ず完成する。
さらに全操作の合法性と完成をリプレイし、失敗時にはV9の安全な
tree/jump補完へ戻す。

## peephole / サイクル圧縮の判断

V9の既存100ケース静的集計では、隣接する同一操作対は0件で、末尾30%の
面積2比率も28.9%（全体26.9%）と偏っていない。このため単純な対合相殺や
「終盤だけ」のpeepholeは期待値が小さく、V10には追加しなかった。

木補完の連続する互いに素な面積2交換については、既存`compress_fallback`が
並べ替えと整列済み長方形への融合を行う。一般の大操作をまたぐ並べ替えや
短サイクル置換は、支持集合の非干渉だけでなく壁を含む合成操作の同値性証明が
必要になるため、今回の安全候補には含めない。次に試す場合は、最大可換区間を
抽出し、同方向・同じ半幅・連続行（または列）の操作だけを1長方形へ融合して、
必ず全リプレイを通す独立候補とする。
