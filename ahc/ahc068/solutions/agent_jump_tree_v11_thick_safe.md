# jump_tree_v11_thick_safe

`agent_jump_tree_v10_zero.rs`を基準に、壁あり入力の4本目の補完候補だけを
V9 `Clearance`から`ThickSafe`へ置換する保守候補。ゼロ壁の決定的C-R-C分岐、
`old`、`V8Edge`、`Distance`の各ガードは変更しない。

## ThickSafe

まずV9と同じ規則で、注目カードが木経路を進む最長`k <= 10`を選び、その
`k`に対する全offsetからClearanceキーで1ラインの操作を決める。その後だけ、
操作と直交する方向へ、注目ラインを含む全配置を最大4ラインまで列挙する。

- 水平ジャンプ`H r c 1 2k`は、高さ2〜4の候補へ拡張する。
- 垂直ジャンプ`V r c 2k 1`は、幅2〜4の候補へ拡張する。
- 長方形の全セルがactive、盤面内、壁なしでなければ採用しない。
- 追加する各ラインについて、全カードの木距離和のdeltaが`<= 0`であることを
  要求する。
- 厚い操作全体のdeltaが元の細い操作以下であることも要求する。
- 合法候補の中では総deltaを最小化し、同値ならライン数が多いものを選ぶ。

半交換は各ラインで独立なので、追加ラインのdeltaの和は厚い長方形のdeltaと
一致する。注目カードを含むラインの始点・半幅は変えないため、注目カードは
従来どおり木経路上の`k`個先へ着地する。確定済みセルはactive検査により
触れない。

## 作業量と安全性

厚さごとの配置数は最大`1 + 2 + 3 + 4 = 10`個で固定。追加ラインのdeltaは
注目ラインの前後3本について一度だけ計算し、固定長配列にキャッシュする。
探索本数、チェックポイント、root/order、壁時計予算はV10から増やさない。

各checkpoint/treeでは従来どおり`old`、`V8Edge`、`Distance`、`ThickSafe`を
独立生成し、完全リプレイを通った最短の補完だけを採用する。最終出力も入力
盤面から再度リプレイされるため、厚い候補が不正または長い場合は既存候補へ
戻る。
